# 隐式 PD vs 显式 PD 控制器

## 先理解 PD 控制本身

不管是隐式还是显式，控制目标是同一个：让关节到达目标位置。

PD 控制的公式：

```
τ = kp × (目标位置 - 当前位置) + kd × (目标速度 - 当前速度)
```

- `kp`（stiffness）：弹簧刚度，位置偏差越大，扭矩越大
- `kd`（damping）：阻尼系数，速度越快，反向制动力越大
- `τ`：最终施加到关节上的扭矩

举个例子：你想让膝关节保持在 0.7 rad，但当前是 0.5 rad。
- 位置差 = 0.2 rad，kp = 150 → 产生 30 Nm 拉向目标
- 同时关节正在快速运动，kd = 5 → 产生阻力防止过冲

**两种模式的区别不在于公式，而在于这个公式在"什么时候、由谁来算"。**

## 隐式 PD（ImplicitActuator）— 当前使用

### 谁来算

PhysX 物理引擎内部。

### 流程

```
策略网络输出 action
       ↓
action 被转换为关节目标位置
       ↓
目标位置 + kp/kd 写入 PhysX 的 JointDrive 属性
       ↓
PhysX 在求解约束时，把 PD 扭矩"隐式"融入物理积分过程
       ↓
最终关节运动（你永远看不到中间的扭矩值）
```

### "隐式"的含义

扭矩 **不是单独算出来的**。PhysX 把 PD 控制当作一个物理约束，和碰撞、重力等一起在求解器里处理。PD 扭矩和物理积分是耦合在一起的——你无法把它从物理仿真中分离出来。

可以类比为：你告诉弹簧"去这个位置"，但弹簧的力不是你算的，而是物理引擎在模拟过程中自然产生的。

### 特点

- **数值稳定**：大时间步长下不会爆炸
- **扭矩不可见**：你不知道实际施加了多少扭矩
- **没有硬裁剪**：物理引擎可能会产生超过电机极限的力
- **和真实电机行为不一致**：真机不是这样工作的

---

## 显式 PD（IdealPDActuator）— 推荐用于 Sim2Real

### 谁来算

Python 代码，在物理仿真步之前。

### 流程

```
策略网络输出 action
       ↓
action 被转换为关节目标位置
       ↓
Python 代码显式计算：
  τ = kp × (目标位置 - 当前位置) + kd × (目标速度 - 当前速度)
       ↓
τ 被 effort_limit 裁剪（超过 120Nm 的部分直接丢弃）
       ↓
裁剪后的 τ 作为外力施加到关节
       ↓
PhysX 只负责物理响应（不参与 PD 计算）
```

### "显式"的含义

扭矩是**单独、明确地算出来的**。每一步仿真开始前，先用公式算出扭矩值，裁剪后交给物理引擎。PD 控制和物理仿真是解耦的。

可以类比为：你自己算出需要多少力，然后把这个力交给物理引擎去执行。

### 特点

- **扭矩完全可控**：你知道施加了多少，可以裁剪
- **和 MuJoCo 一致**：MuJoCo 也是这样工作的
- **和真实电机一致**：真机控制器也是这样算 PD 的
- **适合 sim2real**：仿真和现实的控制逻辑完全对齐

---

## 直观类比

假设你要控制一个门的角度：

| | 隐式 | 显式 |
|---|---|---|
| 做法 | 把门装上一个弹簧，弹簧终点设到目标位置，让物理引擎模拟弹簧效果 | 用手直接推门，你算出需要多大力，施加这个力 |
| 你能看到什么 | 只能看到门在动，看不到弹簧力的大小 | 你知道你用了多少力，可以限制最大力 |
| 超出电机能力 | 弹簧可能产生超出电机极限的力（物理引擎不 care） | 你算出力后检查是否超限，超了就裁掉 |
| 和真机对应 | 真实电机不是弹簧，不能这么对应 | 真实电机控制器就是这样算的 |

## 对 Sim2Real 的影响

### 为什么隐式 PD 导致 sim2sim gap

1. **PhysX 的 PD 行为 ≠ MuJoCo 的 PD 行为**
   - 隐式模式下，PhysX 把 PD 融入约束求解，行为和显式 PD 不同
   - MuJoCo 用的是显式 PD
   - 策略在 PhysX 隐式 PD 下学到的高频补偿，到了 MuJoCo 显式 PD 下完全不适用

2. **扭矩超限行为不同**
   - 隐式：物理引擎可能"软"超限（实际扭矩超过 effort_limit）
   - 显式/真机：硬裁剪，超出部分直接丢弃

3. **高频响应不同**
   - 隐式 PD 在物理积分内部，响应更快更平滑
   - 显式 PD 有一个仿真步的延迟
   - 真实电机也有延迟

### 为什么换成显式 PD 能改善

```
训练环境（Isaac Lab 显式 PD）≈ MuJoCo（显式 PD）≈ 真实电机（显式 PD）
         ↓                              ↓                        ↓
    相同的控制逻辑              相同的控制逻辑            相同的控制逻辑
```

三者的 PD 扭矩计算方式一致，策略学到的行为可以直接迁移。

---

## 迁移成本

改动很小（`magiclab.py` 中 3 处），但需要**从头重新训练**，因为：

- 隐式 PD 下训练的策略已经适应了 PhysX 的特殊动力学
- 显式 PD 的关节响应会不一样（可能更"软"，因为裁剪）
- 策略需要重新学习如何在显式 PD 下平衡
